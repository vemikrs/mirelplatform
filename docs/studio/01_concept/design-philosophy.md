# mirel Studio 設計思想

> mirel Studio の根幹となる設計原則

---

## 1. 基本原則

mirel Studio は、業務アプリケーションを **構造化されたモデルとして定義し、そのモデルをアプリ全体へ一貫適用する** ための環境である。

以下の原則に基づき設計される。

---

## 2. Model-Driven Everything（MDE）

mirel Studio は「モデルを単一の情報源」とする。

- データ構造
- フォーム
- ビジネスロジック
- API
- 権限
- バージョン

これらはすべて "モデル" に基づいて生成・整合される。

UI からデータが生まれるのではなく、**データ構造から UI・ロジック・API が派生する**。

---

## 3. Recursive Domain Structure（再帰的ドメインモデル）

業務を構成する情報は、単純なフィールドの集合ではなく **階層・関係・複合性** を持つ。

mirel Studio は以下を前提とする：

- ドメインは階層構造を持つ
- 複合エンティティを直接扱える
- 再帰的な依存を安全に解決できる
- 一貫性を保ったまま UI とデータへ展開できる

これにより、多階層業務の表現力を確保する。

---

## 4. Consistency First（一貫性の優先）

モデル → UI → ロジック → データ の各要素の間に **整合性の欠落が生じないこと** を最重要とする。

- 型の齟齬が生まれない
- UI の振る舞いとデータ定義が乖離しない
- API がモデルから外れない

業務アプリは "後から直す" ことが必ず発生する。ゆえに、**編集後も壊れない構造** が必要である。

---

## 5. Runtime と Builder の分離

構築フェーズ（Builder）と運用フェーズ（Runtime）は厳密に分離する。

| フェーズ | 役割 |
|---------|------|
| Builder | 設計・試作・改善 |
| Runtime | 安定運用・実利用 |

この分離により、変更が業務に即座に影響しない安全性を確保する。

---

## 6. Release as a Unit（リリース単位での変化管理）

変更は「バージョン」としてまとめて取り扱う。

| 状態 | 説明 |
|------|------|
| Draft | 作業中 |
| Release | 確定したバージョン |
| Deploy | 環境への適用 |
| Rollback | 巻き戻し |

これにより、業務アプリに不可欠な **変更の追跡性・安全性・再現性** を保証する。

---

## 7. Low-Code / No-Code の両立

mirel Studio は以下を両立させる：

- 非開発者によるノーコード構築
- 開発者によるローコード拡張

境界面を曖昧にせず、専門性が価値を発揮できる構造を維持する。

---

## 8. Excel Chaos の終焉

mirel Studio の根底には、**Excel による業務システムの崩壊を止める** という使命がある。

- 再帰性
- 構造化
- 一貫性
- バージョン管理

これらを備えることで、業務の暗黙知と混乱が蓄積される領域を根本から置き換える。

---

## 9. 設計思想のまとめ

mirel Studio は「楽に画面を作るツール」ではない。**業務の構造を扱い、進化に耐えられるシステムとして保持するための基盤** である。

```
Model → UI → Flow → Data → Release
```

この一貫性の上に、mirel Studio のすべての機能が存在する。
