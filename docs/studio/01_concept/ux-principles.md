# mirel Studio UX 原則

> Builder 体験を極力シンプルかつ安全にするための操作性ガイドライン

---

## 概要

mirel Studio は Builder 体験を極力シンプルかつ安全にするため、以下の UX 原則に基づいて設計される。

---

## 原則 1：モデルを中心とする

すべての画面・動作は「モデル」に基づく。

- Modeler が唯一の SSOT（Single Source of Truth）
- Form/Flow/API は派生物
- UI 上の情報はモデルの構造を反映する

**結果：**
- 認知負荷が大幅に低減
- フォームやフローの破綻を防止

---

## 原則 2：必要な情報だけを提示する

Builder は少量の情報から判断できるようにする。

**例：**
- プロパティパネルは「選択した項目」だけ表示
- 一度に表示する情報は最小限
- 詳細は折りたたみ構造に集約

---

## 原則 3：予測可能であること

操作結果が常に予測できるようにする。

- 削除は確認ダイアログ
- モデル変更は影響箇所を一覧提示
- フロー条件は型安全にチェック

---

## 原則 4：可視性の確保（Visibility）

現在の状態が明確に見えるようにする。

| 領域 | 表示内容 |
|------|---------|
| 左ナビ | 機能構造 |
| 中央 | 編集対象 |
| 右ペイン | 詳細プロパティ |
| Release Center | 差分の一覧化 |

---

## 原則 5：Undo/Redo と安全な編集

Builder 作業中は破壊的変更が起きないようにする。

- Draft はいつでも巻き戻し可能
- Release の生成までは「仮の状態」
- 環境（Dev/Stg/Prod）と編集環境を分離

---

## 原則 6：一貫性（Consistency）

UI・色・レイアウト・命名規則は統一する。

- モデル名 ＝ API 名 ＝ フォーム名
- フィールド順序 ＝ 初期表示順
- Flow ノードの配置順 ＝ 実行順

---

## 原則 7：ユーザーに選ばせすぎない

選択肢が多いと Builder の負荷が増えるため、設計上の判断はシステムが可能な限り自動化する。

**例：**
- ウィジェットの自動割り当て
- モデル変更による同期処理
- Flow の型安全チェック

---

## 原則 8：視覚的階層（Visual Hierarchy）

構造の複雑さは視覚設計で調整する。

- 文字サイズ
- アイコン
- セクション
- タブ／折りたたみ

---

## 目的

- Builder のストレスを最小限に
- LCAP 特有の "複雑化" を抑制
- モデル駆動の UI/UX を統一的に実装
