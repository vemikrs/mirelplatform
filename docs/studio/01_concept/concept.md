# mirel Studio コンセプト

> mirel Studio の製品コンセプトと差別化ポイント

---

## 1. Positioning（位置付け）

mirel Studio は、**業務アプリケーションの構造・表現・振る舞い・進化** を一体的に扱うための統合環境である。

一般的な LCAP が「フォームを置けばアプリができる」という発想に基づくのに対し、mirel Studio は **「業務モデルを記述すればアプリが成立する」** という逆向きの思想を採る。

---

## 2. Core Message（核心）

> **mirel Studio は、業務構造を "モデル" として編成し、そのモデルをシステム全体に一貫適用するための設計環境である。**

UI ではなく、構造が源泉になる。この一点が、既存のローコード／ノーコードと決定的に異なる。

---

## 3. Key Differentiators（差別化ポイント）

### 3.1 業務を「再帰的な構造」として扱う

- 階層データ
- 複合エンティティ
- 部分構造の再利用
- 多階層バインディング

これらを前提としたモデル化が可能。既存 LCAP が苦手とする **複雑な業務体系をそのまま表現できる**。

### 3.2 Model-Driven Everything（MDE）

モデルがすべての源泉となり、以下が一貫した整合性で自動生成・連携される：

- フォーム
- フロー
- API
- データ操作
- バージョン管理

### 3.3 変更の蓄積を "秩序として" 扱える Release Center

業務アプリは必ず変化する。mirel Studio はその変化を「破壊要因」ではなく **「秩序を育てるプロセス」** として扱う。

- Draft / Release / Deploy
- 差分検出
- 影響範囲の明確化
- ロールバック可能

変更が積み重なるほど、モデルは強くなる。

### 3.4 Builder と Runtime の厳密な分離

誤編集・事故反映・現場混乱を防ぐため、設計と運用を明確に切り分ける。

| 環境 | 役割 |
|------|------|
| mirel Studio | 設計 |
| Runtime | 安定稼働 |

この構造はエンタープライズに耐える唯一の方式である。

### 3.5 エンジニア視点の NCAP（整合・再利用・依存制御）

ノーコード向けの "平面的な設計" ではなく、業務アーキテクトが重要視する以下の原則を満たす：

- 情報の単一性（SSOT）
- 依存方向の制御
- 型安全性
- 再利用可能なモデル単位
- 修正後も壊れない一貫性

**"現場向けに簡略化したツール" ではなく、アーキテクチャ原則に基づく本物の NCAP** として機能する。

---

## 4. Component Architecture（コンポーネント全体像）

```
mirel Studio
├── Modeler        … 業務構造を記述する中心機能
├── Form Designer  … UI の表現
├── Flow Designer  … 業務ロジックと連携の定義
├── Data Browser   … データの観測と軽微な操作
└── Release Center … 変更の統制と環境管理
```

これら 5 要素は "モデル" を中心に結びついている。

| コンポーネント | 正式名称 | 役割 |
|---------------|---------|------|
| Modeler | mirel Studio Modeler | 業務モデルの定義 |
| Form Designer | mirel Studio Form Designer | UI レイアウトの設計 |
| Flow Designer | mirel Studio Flow Designer | ビジネスロジックの設計 |
| Data Browser | mirel Studio Data Browser | データの閲覧・操作 |
| Release Center | mirel Studio Release Center | リリース管理 |

---

## 5. Vision（ビジョン）

> **複雑な業務が持つ構造と秩序を、そのまま扱える形でモデル化し、システムとして安定的に維持できる世界を実現する。**

業務の知識は日々変化し、Excel や属人化したファイル群の中で劣化していく。

mirel Studio はそれらを **構造化された業務モデルとして外在化し、改善の資産として蓄積する**。

業務の論理とシステムの実装が乖離しない。その状態を永続的に保つための基盤が mirel Studio である。

---

## 6. Summary（要約）

- mirel Studio は CRUD ツールではなく **業務構造を扱うモデル駆動の設計環境**
- 再帰的ドメイン構造と一貫性管理が最大の特徴
- フォーム／フロー／データ／公開がすべてモデル従属
- 変更の蓄積に耐え、むしろ組織の知識として成長させる
- アーキテクトの原則と、現場の利用性を両立させる NCAP
